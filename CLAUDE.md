# Open Data Playground ‚Äî project context

Analysis of traffic accidents (DTP) in Serbia from the data.gov.rs open data portal, 2015‚Äì2026.

## Data

- `data/dtp_2015.xlsx` ‚Ä¶ `data/dtp_2026.xlsx` ‚Äî raw per-year source records. Columns (no header row): id, region, municipality, datetime (`DD.MM.YYYY,HH:MM`), lon, lat, severity (`Sa mat.stetom` / `Sa povredjenim` / `Sa poginulim`), category, description.
- `data/all_records.parquet` ‚Äî all records, `dt_raw`/`severity`/`dt` only (region/municipality/coords were dropped when this was first built ‚Äî re-derived separately for the geo pipeline below).
- `data/yearly_summary.csv` ‚Äî per-year summary.
- `data/{yearly,monthly,weekly,daily}.json` ‚Äî nationwide precomputed aggregates, fields: `mat` (property damage only), `inj` (injuries), `fat` (fatalities), `total`. Weekly has `iso_year`/`iso_week`, daily has `date`.
- `data/geo/index.json` + `data/geo/region/{0..26}.json` + `data/geo/muni/{0..161}.json` ‚Äî same yearly/monthly/weekly/daily aggregate shape, scoped to one region or municipality (27 regions, 162 municipalities, each municipality belongs to exactly one region). Built from the raw xlsx region/municipality columns. Index files are keyed by array position, not name ‚Äî filenames are just the index.
- `data/weather_raw/{city}.json` ‚Äî raw Open-Meteo daily archive responses for 8 representative cities (Beograd, Novi Sad, Nis, Kragujevac, Subotica, Uzice, Zrenjanin, Kraljevo), one fetch per city for the whole 2015-01-01..2026-05-31 range (`archive-api.open-meteo.com/v1/archive`, no API key). This is a one-time historical fetch ‚Äî don't re-fetch, the data doesn't change.
- `data/weather_{yearly,monthly,weekly,daily}.json` ‚Äî the 8 cities' daily values simple-averaged per day (temp = mean, precip/snow/wind = summed/maxed as appropriate), then bucketed at all 4 granularities to match the accident aggregates 1:1 on `(year)` / `(year, month)` / `(iso_year, iso_week)` / `(date)`. All four are derived from the same cached `weather_raw/` files ‚Äî regenerating them needs no new API calls.
- Data spans 2015-01-01 ‚Ä¶ 2026-05-31. **2015 and 2026 are partial years**: 2015 collection ramped up mid-year (sharp jump from September 2015), 2026 cuts off at May.
- Everything under `data/` is duplicated into `app/data/` ‚Äî that's the copy the dashboard actually fetches. Re-copy manually after regenerating anything in `data/`.

## App (`app/`)

Static site, no build step: `index.html` + `style.css` + `app.js`, data loaded via `fetch('data/*.json')`.

- All UI copy is **in English** (chat conversation stays Russian, app is English). Code comments in English too.
- Run: `cd app && npm run dev` (alias for `python3 -m http.server 8080`) ‚Üí http://localhost:8080.
- Deploy: any static host with root = `app/`, no build command (see `app/README.md`). For platforms that only run a Python/WSGI process (e.g. hosts requiring a `Procfile`/`gunicorn`), `wsgi.py` at the repo root is a thin Flask wrapper serving the same `app/` files unchanged (`static_folder=app/`, `/` ‚Üí `index.html`) ‚Äî no app logic lives there, it's purely a hosting-compatibility shim. Run with `gunicorn wsgi:app` (see `Procfile`/`requirements.txt`).

### Chart / range picker
- Default view on load is **Week**, not Year (per user preference). Year, Month, Week render as a stacked bar chart. Day is dynamic: bars when the visible window is ‚â§150 points (e.g. a narrowed custom range), area when wider (e.g. "All time" ‚âà 4169 points is too dense for bars) ‚Äî see `DAY_BAR_THRESHOLD` in app.js.
- Date-range picker: GA4-style trigger button + popover ‚Äî named presets (last 7/30 days, last month/quarter/year/3y/5y, this year, all time, custom), typed `DD.MM.YYYY` inputs, a click-to-select calendar, and the granularity switch, all staged and committed via Apply. Presets anchor to the dataset's max date (2026-05-31), not real wall-clock time.
- Region/municipality filter: two styled `<select>`s next to the range trigger. Selecting either lazy-fetches the matching `data/geo/{region,muni}/{idx}.json` bundle (cached in memory after first fetch) and swaps it in as the active data source; "All regions"/"All municipalities" reverts to the nationwide files with zero extra fetch. Sparse municipalities are zero-filled (`densify()` in app.js) against the nationwide period list so area/bar charts don't silently skip zero-accident periods.
- Holiday markers: small dots above the bar/area top on Week and Day views only (Year/Month are too coarse), for a static hand-built list of Serbian public holidays 2015‚Äì2026 (`HOLIDAYS` in app.js; Orthodox Easter dates computed via the Julian/Meeus algorithm, not hand-typed). Native SVG `<title>` gives the hover name. Legend item always visible with a **click-to-open** `(?)` hint (not hover ‚Äî a hover-only `title` tooltip wasn't discoverable enough) explaining the week/day-only scope; dimmed (not hidden) on Year/Month so the hint stays reachable. The hint popover uses the same `composedPath()` outside-click-close pattern as the date-range popover.
- Weather overlay: a proper on/off switch (`#weatherCheckbox`, not a text-swapping button) next to "Table". Lazy-fetches all 4 `data/weather_{yearly,monthly,weekly,daily}.json` files on first activation and works at **every** granularity, joined by the same period key as the accident points (`weatherKeyForPoint`/`weatherRowKey` in app.js). Renders as two lines on their own right-hand scale: temperature (solid) and precipitation (dashed) ‚Äî precipitation was originally a low-opacity area fill but that was too subtle to read against the bars, changed to a line per user feedback. Tooltip appends temp/precip/snow/wind rows when weather is active.
- KPI number tiles were removed per user request (hidden via `hidden = true`, not deleted from the DOM, in case they come back) ‚Äî only the Absolute/Per-1,000-vehicles toggle itself stays visible. That toggle now does something real: it re-scales the *chart itself* (bars/area/axis/tooltip/table), not just the old hidden tiles ‚Äî see `toDisplayPoints()` in app.js, which converts each point's mat/inj/fat/total into a per-1,000-vehicles rate before it reaches the renderer. The per-1,000 series (`FLEET_ESTIMATE`/`fleetForYear()` in app.js) is **not** an official year-by-year RZS series ‚Äî only 2015 (1,833,219) and 2023 (2,389,105) passenger-car counts are confirmed; everything else is geometric interpolation/extrapolation at the constant ~3.366%/year CAGR those two imply. RZS's real per-year table lives behind an interactive form (`data.stat.gov.rs/Home/Result/150201`) that couldn't be scraped programmatically ‚Äî replace `FLEET_ESTIMATE` if a real series is ever pulled from there. The `norm-note` caption discloses this whenever per-1,000 mode is active.

### Known bugs fixed along the way (don't reintroduce)
- `computeTicks`'s tick-building loop could stop one step short of the real max (bars overflowing the SVG, overlapping the legend above) ‚Äî fixed by looping until the last tick is ‚â• maxVal, not by a fixed iteration bound.
- `niceStep` could round to a sub-1 step for small maxVal (e.g. a single quiet day for a small municipality), which rounds to 0 every iteration and spins `computeTicks`'s loop forever (`Invalid array length`). Fixed by flooring the step at 1 ‚Äî counts are always integers.
- The popover's "click outside to close" listener used `contains(e.target)`, which breaks when the click handler rebuilds the DOM (e.g. picking a preset) mid-bubble and detaches `e.target` before the outer listener runs. Fixed with `composedPath()`, which is fixed at dispatch time.
- The calendar used to rebuild all 42 day cells on every `mouseenter` for the hover-preview highlight ‚Äî thrashed the DOM under the cursor. Split into a one-time grid build (on month change) and a cheap class-only highlight update (on hover).
- A general `svg { width:100%; height:auto }` rule (added for the chart's responsiveness) also hijacked the small icon svgs in the trigger buttons, blowing them up to fill their container. Scoped to `#chart` only.
- Recurring gotcha: any element toggled via the `hidden` JS/HTML attribute needs an explicit `[hidden] { display: none; }` rule if its base class also sets `display` (flex/block/etc) ‚Äî the class's `display` otherwise beats the browser's default `[hidden]` styling and the element stays visible with `hidden` set. Hit this three times: `.empty-msg`, `.stats-row`/`.norm-toggle`, `.legend-item` (the last one silently broke the weather legend items). Check for this pattern first whenever a `hidden`-toggled element "isn't hiding."
- `addMonthsClamped` used `Date.setUTCMonth()` directly, which overflows into the next month when the target month is shorter (e.g. subtracting 1 month from May 31 lands on May 1, not April 30). Fixed by clamping the day-of-month to the target month's real length.
- `computeTicks`/`niceStep` originally floored every step at a whole number (`Math.max(1, ...)`, assuming counts are always integers) ‚Äî correct for absolute mode but wrong once per-1,000-vehicles rates were added, since a weekly rate maxing out around 0.3 would get an axis that floors to 0/1 and makes every bar look flat. Fixed by moving the integer-flooring behind an `integerSteps` param (`computeTicks(maxVal, targetCount, integerSteps)`), passed as `normMode !== 'per1000'` at both call sites, and by not rounding inside the tick-accumulation loop (plain float addition; rounding only happens for display, in `tickText`).
- A `Math.max(...values, 1)` fallback (meant only to avoid a divide-by-zero on all-zero data) was silently forcing the chart's y-axis max to at least `1` even when the real max was a small decimal like `0.27` (per-1,000 mode). Fixed to `Math.max(...values) || 1`, which only kicks in when the real max is exactly `0`.
- Real Safari (not reproducible in Chromium or Playwright's own WebKit build) painted the SVG chart on top of the holiday/norm hint popovers despite them having a higher `z-index` and being later in paint order ‚Äî a known category of WebKit bug where a heavy `<svg>` (ours has ~1800 path elements at Week granularity) gets promoted to its own compositing layer that ignores normal stacking rules against sibling positioned elements. No CSS-only fix (explicit `position`/`z-index` on `#chart`) resolved it. Fixed structurally instead: `wireHint()` in app.js reparents the popover to `document.body` and switches it to `position: fixed` with JS-computed coordinates (`getBoundingClientRect` on the trigger button) ‚Äî this removes it from the chart's ancestor tree entirely so there's no shared stacking context left to fight over. If another floating element (tooltip, dropdown) ever gets visually trapped under the chart, reach for this pattern rather than tweaking z-index further.
- A `.norm-note[margin: -20px ...]` rule was written when the caption sat directly under the (visible) stat tiles and needed to tuck up under them. Once the tiles were hidden, that negative margin pulled the note up into the toggle buttons above it instead. Fixed by zeroing the top margin now that there's nothing above it to tuck under ‚Äî revisit if the tiles ever come back.
- `Edit`'s `replace_all` matches the literal string given ‚Äî `let currentRange = 'year';` (the declaration) and `currentRange = 'year';` (the reassignment inside `init()`, no `let`) are *different strings* and only the first got caught by a `replace_all` on the `let`-prefixed form, silently leaving the actual runtime default unchanged. When changing a variable's default, grep for *all* assignment sites (`grep -n "varName = "`), not just the declaration.

## Open findings / analysis threads

- Seasonality (averaged 2016‚Äì2025, accidents/day): trough in February (~82), gradual rise to an October peak (~104), a dip in November, a secondary rise in December (~101).
- Property-damage-only (minor) crashes peak in October/December; fatalities, by contrast, peak in summer (July‚ÄìSeptember) and decline toward winter ‚Äî i.e., the pre-winter rise in accident count isn't accompanied by a rise in severity.
- Seasonality hypotheses, now partly testable in-app via the weather overlay (any granularity) and holiday markers (Week/Day): autumn weather/black ice, shorter daylight hours, a summer fatality peak (motorcycle season, holiday travel, speed), December holiday traffic, a possible year-end reporting artifact.
- A self-contained single-file version of an early build (data hardcoded into JS, no region/holiday/weather features) lives in scratchpad and was published via the Artifact tool for a shareable preview link. It's stale relative to `app/` and not the source of truth.
